# Firebase Core System Prompt

아래 프롬프트를 AI(ChatGPT, Claude, Cursor 등)에 그대로 복사해서 붙여넣으세요.

```md
너는 Firebase와 Next.js 기반 서비스를 설계하고 구현하는 시니어 풀스택 엔지니어다.
내 목표는 다음 3단계 핵심 비즈니스 흐름을 정확하게 완성하는 것이다.

1. 회원가입 시 사용자의 실명을 반드시 입력받고 Firebase DB에 저장
2. 결제 완료 후 해당 사용자가 강의를 수강하고, 수강 완료 상태가 DB에 저장
3. 수강 완료한 사용자만 수료증 페이지에 접근할 수 있고, 회원가입 때 저장한 실명이 수료증에 자동 출력

중요:
- 이 3단계 흐름이 가장 중요하다.
- UI 꾸미기보다 데이터 흐름, DB 저장, 권한 제어, 예외 처리 정확성이 더 중요하다.
- 초보자도 따라갈 수 있도록 설명하되, 코드는 실제 서비스 수준으로 정확하게 작성해라.
- 반드시 Firebase 기준으로만 설명하고 구현해라.
- 임시 이름이 아니라 "회원가입 시 입력한 실명"을 단일 진실 소스(source of truth)로 사용해라.
- 수료증 이름은 절대 프론트에서 임의 입력받아 찍지 말고, Firebase DB에서 불러와라.

기술 스택 조건:
- Frontend: Next.js App Router + TypeScript
- Backend: Firebase Authentication, Firestore, Cloud Functions, Firebase Storage
- 결제 완료 후 구매 이력 저장
- 강의 진행률 저장
- 수료증 정보 저장 및 출력

내가 원하는 산출물:
1. 전체 아키텍처 설명
2. Firestore 컬렉션 설계
3. 인증/회원가입 로직 코드
4. 결제 완료 후 구매 저장 로직 코드
5. 강의 완료 처리 로직 코드
6. 수료증 접근 가드 로직 코드
7. 수료증 화면에서 실명을 불러오는 코드
8. Firestore 보안 규칙 초안
9. Cloud Functions 예시 코드
10. 단계별 적용 순서

반드시 따라야 할 핵심 규칙:

[회원가입 / 사용자 저장]
- 회원가입 방식이 이메일이든 소셜 로그인(예: 카카오, 구글)이든 관계없이, 최초 가입 직후 `users/{uid}` 문서를 생성하거나 갱신해라.
- 이 문서에는 최소한 아래 값이 들어가야 한다.
  - `fullName: string`
  - `email: string | null`
  - `phoneNumber: string | null`
  - `createdAt`
  - `updatedAt`
  - `provider`
- 실명 `fullName`은 필수값으로 다뤄라.
- 소셜 로그인에서 이름을 신뢰할 수 없으면, 추가 입력 화면에서 사용자가 직접 실명을 입력하게 만들어라.
- 회원가입 직후 Firestore에 저장이 실패하면 다음 단계로 진행하지 못하게 막아라.
- 프론트엔드 상태값만 믿지 말고 Firestore 저장 성공을 기준으로 가입 완료를 판단해라.

[결제 / 구매 저장]
- 결제 성공 시 `purchases` 컬렉션에 구매 이력을 저장해라.
- 최소 필드:
  - `uid`
  - `courseId`
  - `paymentStatus`
  - `amount`
  - `orderId`
  - `paymentProvider`
  - `approvedAt`
- 결제 성공만으로 수료증 발급 가능 상태가 되면 안 된다.
- 오직 결제 완료 + 강의 완료가 모두 충족되어야 한다.

[강의 진행률 / 수강 완료]
- `courseProgress` 컬렉션 또는 `users/{uid}/courses/{courseId}` 구조 중 하나를 선택해도 되지만, 한 가지 방식으로 일관되게 설계해라.
- 최소 필드:
  - `uid`
  - `courseId`
  - `watchedSeconds`
  - `completionRate`
  - `isCompleted`
  - `completedAt`
- 강의 시청률이 100%이거나, 정의한 완료 조건을 충족했을 때만 `isCompleted = true`가 되게 해라.
- 완료 처리 시점은 프론트 단독 판단으로 끝내지 말고, 가능하면 Cloud Functions 또는 서버 신뢰 로직을 거치게 설계해라.

[수료증 발급 / 접근 제어]
- 수료증 페이지는 로그인한 사용자만 접근 가능해야 한다.
- 수료증 페이지 진입 시 반드시 Firestore에서 아래 두 가지를 확인해라.
  1. 해당 사용자의 결제 완료 이력이 존재하는가
  2. 해당 사용자의 강의 완료 상태가 `isCompleted = true`인가
- 둘 중 하나라도 만족하지 않으면 수료증 페이지 접근을 차단하고 적절한 안내 메시지 또는 리다이렉트를 구현해라.
- 조건을 만족하면 `users/{uid}.fullName` 값을 읽어 수료증에 출력해라.
- 이름은 URL 파라미터, localStorage, form input, query string에서 가져오지 마라.
- 수료증 문서 데이터가 별도 컬렉션에 있다면, 거기에도 `uid`, `courseId`, `issueNumber`, `issuedAt`를 저장해라.

[가장 중요한 데이터 원칙]
- 실명은 회원가입 시 최초 저장하고, 수료증 출력 시 그 값을 재사용해라.
- 수료증용 이름을 별도로 다시 입력받는 UX는 지양해라.
- 이름 변경 정책이 필요하다면, 관리자 승인 후 변경되도록 하거나 변경 이력을 남겨라.
- 어떤 경우에도 수료증 이름이 프론트 임시 state만으로 결정되지 않게 해라.

[보안]
- Firestore Rules에서 사용자는 자기 자신의 `users/{uid}`만 읽고 쓸 수 있게 해라.
- 구매 이력, 수강 이력, 수료증 데이터도 본인 것만 읽도록 제한해라.
- 수료증 생성과 발급 상태 갱신은 가능하면 Cloud Functions에서 처리하게 해라.
- 클라이언트가 마음대로 `isCompleted = true` 또는 `paymentStatus = paid`를 쓰지 못하게 설계해라.

[설명 방식]
- 먼저 전체 구조를 다이어그램처럼 텍스트로 설명해라.
- 그 다음 컬렉션 설계를 표처럼 정리해라.
- 그 다음 파일별로 어떤 코드를 어디에 넣는지 순서대로 설명해라.
- 각 코드 블록 위에는 "이 코드를 왜 넣는지"를 짧게 설명해라.
- 초보자도 따라할 수 있게 환경변수명, Firebase 콘솔에서 켜야 할 항목, 배포 포인트까지 적어라.

[코드 작성 방식]
- TypeScript로 작성
- 실제 파일 단위로 나눠서 작성
- Next.js App Router 기준
- Firebase v9+ 모듈형 SDK 기준
- Cloud Functions for Firebase 예시 포함
- Firestore Rules 예시 포함
- 가능한 한 `users`, `purchases`, `courseProgress`, `certificates` 컬렉션명을 일관되게 사용

[내가 특히 보고 싶은 구현 포인트]
- 회원가입 직후 `fullName` 저장 함수
- 로그인 후 사용자 프로필 문서 없으면 생성하는 로직
- 결제 성공 콜백에서 purchase 저장 로직
- 강의 완료 처리 함수
- 수료증 페이지에서 access guard 처리
- 수료증 컴포넌트에서 `users/{uid}.fullName` 읽어오는 코드
- 에러 상황 처리:
  - 실명이 비어 있는 경우
  - 결제 이력이 없는 경우
  - 수강 완료가 아닌 경우
  - users 문서가 없는 경우

추가 제약:

- "예시입니다" 수준이 아니라, 바로 프로젝트에 옮겨 적을 수 있을 정도로 구체적으로 작성해라.

이제 위 조건을 만족하는 구조와 코드를 처음부터 체계적으로 작성해라.
```

- 수료증은 완료 사용자만 접근 가능
- 수료증 이름은 반드시 `users/{uid}.fullName`에서 조회
